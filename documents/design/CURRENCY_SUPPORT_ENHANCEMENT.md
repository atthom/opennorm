# Enhanced Currency Support with Compile-Time Code Generation

**Status:** Implementation Guide  
**Date:** 2026-05-12  
**Purpose:** Document the enhanced currency support approach using compile-time code generation

---

## Problem Statement

The initial dimensional analysis implementation encountered a limitation with currency units:

```julia
# This fails when called inside a function:
@dimension EURDimension "EUR" EURDimension
@refunit EUR "EUR" EURUnit EURDimension false
```

**Error:** `Global method definition needs to be placed at top level`

Julia requires `@dimension` and `@refunit` macros to be evaluated at the top level of a module, not inside functions. This prevents dynamic currency registration at runtime.

---

## Solution: Compile-Time Code Generation

Instead of trying to register currencies dynamically, we generate a Julia source file containing the currency definitions, which can then be included at the top level.

### Architecture

```
Taxonomy (markdown)
    ↓
extract_units_from_taxonomy()
    ↓
generate_currency_units_file()
    ↓
generated_units.jl (top-level definitions)
    ↓
include("generated_units.jl")
    ↓
Fully registered currency dimensions
```

## Implementation

### 1. Code Generation Module

**File:** `src/julia/currency_codegen.jl`

Key functions:

- `generate_currency_units_file(unit_defs, output_path)` - Generate Julia source file
- `generate_and_load_currency_units(unit_defs)` - Convenience wrapper
- `precompile_currency_units(taxonomy_path)` - Build step integration

### 2. Generated File Structure

Example output for EUR and USD:

```julia
# Auto-generated currency unit definitions
# Generated at: 2026-05-12T15:58:00
# DO NOT EDIT - This file is automatically generated

using Unitful
using Unitful: @dimension, @refunit, @unit

# Currency Dimensions
# Each currency gets its own dimension to prevent mixing

# Dimension for EUR
@dimension EURDimension "EUR" EURDimension

# Reference unit for EUR
@refunit EUR "EUR" EURUnit EURDimension false

# Dimension for USD
@dimension USDDimension "USD" USDDimension

# Reference unit for USD
@refunit USD "USD" USDUnit USDDimension false

# Register currency units with Unitful
const localunits = Unitful.basefactors
Unitful.register(localunits, EUR)
Unitful.register(localunits, USD)

# Export all defined units
export EUR, USD
```

### 3. Usage Patterns

#### Pattern A: Build-Time Generation (Recommended)

```bash
# Step 1: Generate units file as build step
julia -e '
include("src/julia/unit_system.jl")
include("src/julia/currency_codegen.jl")
using CommonMark

# Parse document and extract units
ast = CommonMark.Parser()(read("documents/articles/CGI.Art.156.opennorm.md", String))
objects = parse_taxonomy(ast, Object, "cgi.art156")
unit_defs = extract_units_from_taxonomy(objects)

# Generate units file
generate_currency_units_file(unit_defs, "src/julia/generated_units.jl")
'

# Step 2: Include in main module
# In src/julia/opennorm.jl:
include("generated_units.jl")
```

#### Pattern B: Development-Time Generation

```julia
# In REPL or script:
include("src/julia/opennorm.jl")

# Parse document
doc = parse_document("documents/articles/CGI.Art.156.opennorm.md")

# Extract units
unit_defs = extract_units_from_taxonomy(doc.objectTaxonomy)

# Generate file
include("src/julia/currency_codegen.jl")
generate_currency_units_file(unit_defs, "src/julia/generated_units.jl")

# Restart Julia and include the generated file
```

#### Pattern C: Makefile Integration

```makefile
# Makefile

generate-units:
	julia --project=. -e '\

```

## Advantages

### 1. Full Currency Dimension Support

Each currency gets its own dimension, preventing accidental mixing:

```julia
1.0EUR + 1.0USD  # ❌ DimensionError - cannot add different currencies
1.0EUR × 2.0     # ✅ 2.0EUR - scalar multiplication works
1.0EUR / 1.0EUR  # ✅ 2.0 (dimensionless) - division works
```

### 2. Type Safety

```julia
# Procedure validation catches errors:
*RevenuEUR* = *RevenuUSD*  # ❌ Type mismatch: USD ≠ EUR
*RevenuEUR* = *RevenuUSD* × *TauxChange*  # ✅ If TauxChange is USD/EUR
```

### 3. Compile-Time Verification

All currency operations are checked at compile time, not runtime.

### 4. Clean Separation

- **Taxonomy** = Source of truth for units
- **Code generation** = Build-time transformation
- **Generated file** = Compile-time definitions
- **Runtime** = Type-safe operations

## Limitations and Trade-offs

### Current Limitations

1. **Requires Build Step**: Can't dynamically add currencies at runtime
2. **File Generation**: Adds generated file to source tree
3. **Restart Required**: Changes to taxonomy require regeneration and restart

### Acceptable Trade-offs

These limitations are acceptable because:

1. **Currencies are stable**: Tax codes don't add new currencies frequently
2. **Build steps are normal**: Most projects have build/codegen steps
3. **Type safety worth it**: Compile-time checking prevents runtime errors

### Future Enhancements

1. **Watch Mode**: Auto-regenerate on taxonomy changes
2. **Incremental Updates**: Only regenerate if taxonomy changed
3. **Multiple Taxonomies**: Merge units from multiple documents
4. **Unit Conversions**: Generate conversion factors between currencies

## Comparison with Simplified Approach

### Simplified (Current in unit_system.jl)

```julia
# Treats all currencies as dimensionless
function register_units!(unit_defs::Vector{UnitDefinition})
    for unit_def in unit_defs
        if unit_def.unit_type == "Currency"
            @warn "Currency $(unit_def.symbol) registered as dimensionless"
            # No actual dimension created
        end
    end
end
```

**Pros:**
- Works immediately
- No build step needed
- Simple implementation

**Cons:**
- ❌ No type safety for currencies
- ❌ Can accidentally mix EUR and USD
- ❌ EUR × EUR not caught as error

### Enhanced (Code Generation)

```julia
# Each currency gets proper dimension
@dimension EURDimension "EUR" EURDimension
@refunit EUR "EUR" EURUnit EURDimension false
```

**Pros:**
- ✅ Full type safety
- ✅ Prevents currency mixing
- ✅ Catches dimensional errors

**Cons:**
- Requires build step
- More complex setup
- Generated file in source tree

## Conclusion

The compile-time code generation approach provides full currency dimension support while working within Julia's constraints. It requires a build step but delivers complete type safety for currency operations.

**Recommendation:** Use the simplified approach for initial development, then migrate to code generation when currency type safety becomes important.
